## 1. WHAT IS THIS SYSTEM (for Opus clone context)

**TypeScript monorepo** at `signal-beacon-1/`:
- `packages/scanner` — Node.js service; tracks Polymarket odds, detects whale trades, generates trading signals, stores everything in SQLite
- `packages/api` — Express server on port 3100; REST API for dashboard + HA notifications
- `packages/dashboard` — Vite/React frontend on port 8080
- `scripts/continuous-scan.js` — The main runtime loop (10-min scan, 30-min tweet collect, 6h market refresh, 24h cleanup)

**Core data flow:**
```
Polymarket Gamma API
  → MarketDiscoverer (filter by ontology keywords, 3500+ markets tracked)
  → OddsTracker (snapshot odds every 10 min, detect delta changes)
  → WhaleDetector (check Polymarket trades API per changed market)
  → SignalGenerator (map market → assets via ontology, dedup 4h window)
  → AlertDispatcher (HA push if conf≥65, market open, delta≥15%)
```

**Key files that cannot be read by a fresh clone:**
- `data/polysignal.db` — 190MB SQLite (190MB main + 122MB WAL ⚠️). Contains all signals, snapshots, whale events, tweet accounts.
- `data/ontology.json` — 24 individual stock entries defining keyword→asset→polarity mappings. This IS in the repo.
- `.env` — Credentials: `AVANZA_USERNAME/PASSWORD`, `HA_URL/HA_TOKEN/HA_NOTIFY_SERVICE`, `PUBLIC_URL`. NOT in repo.
- `data/scan-log.txt` — Runtime log (see section 3 for current state)

## 3. IDENTIFIED BUGS / PROBLEMS

### BUG A: Bull + Bear simultaneously for same asset (Evolution Gaming, Spotify) ❌
**Symptom**: User sees `BULL Evolution Gaming` and `BEAR Evolution Gaming` as separate signals at the same time.

**Root cause**: Both assets have `"default_polarity": "context_dependent"` in `data/ontology.json`:
```json
// gaming-evolution
"correlation_logic": {
  "default_polarity": "context_dependent",
  "explanation": "iGaming legalization is BULLISH for Evolution. Strict gambling regulation or market bans are BEARISH. Requires judgment."
}
// tech-spotify
"correlation_logic": {
  "default_polarity": "context_dependent",
  "explanation": "Higher streaming royalty requirements are BEARISH. Market share growth is BULLISH. Requires judgment."
}
```

**What the code does** (`generator.ts:100-109`):
```ts
if (mapping.polarity === 'context_dependent') {
  return [
    this.createSignal(..., 'bull'),
    this.createSignal(..., 'bear'),
  ];
}
```
This is **intentional by design** — creates both signals capped at confidence 40, flagged `requires_judgment: true`. But the user experiences it as "the system is confused."

**Why it's bad UX**: The user sees contradictory signals and doesn't know which to act on. The confidence cap at 40% and `requires_judgment` flag aren't prominent enough in the dashboard.

**Fix option 1 (quick)**: Change Evolution Gaming polarity to `direct` (iGaming legalization → odds UP → BULL EVO). The market for "will US online gambling be legal?" going UP means BULLISH for Evolution.
**Fix option 2 (proper)**: Show `context_dependent` signals as a SINGLE card with two-sided layout, not two separate signals. The dashboard needs to render them differently.
**Fix option 3**: Add noise filter — only emit `context_dependent` signals when there's also whale detection or confidence > 35% (most are trivially low).

### BUG B: Tweet collection completely broken — `collected=0 errors=0` every cycle ❌
**Symptom**: Every 30-min tweet collection cycle logs:
```
[cycle 1] Collecting tweets...
[cycle 1] Tweets: accounts=26 collected=0 errors=0
```
**Root cause**: Nitter is dead. All three Nitter instances in `collector.ts:3-7` are down or blocking scrapers:
```ts
const NITTER_INSTANCES = [
  'https://nitter.privacydev.net',    // likely down
  'https://nitter.poast.org',         // likely down
  'https://nitter.woodland.cafe'      // likely down
]
```
The `fetchNitterRss()` method catches all errors silently (`catch { continue }`). If all Nitter instances 404 or timeout, it falls back to `fetchRssHub()`. RSSHub (`https://rsshub.app/twitter/user`) also fails silently.

**Impact**:
- `tweet_snapshots` table has 0 rows
- `TweetIntelligenceProcessor.processTweetBatch()` returns early (`tweets.length < 5` check at processor.ts:35)
- No tweet insights stored in `intelligence_memory`
- Morning briefing has no tweet context
- The entire tweet intelligence layer is an elaborate no-op

**The `strategic-seed.json` issue**: Codex generated a 880KB JSON file with 2200 Twitter accounts. This seeds `tweet_accounts` table with 2200 rows on startup. But seeding accounts without being able to collect tweets is pointless. The table gets 2200 accounts, `collect_enabled=true` for ~26 core ones, but all fetch attempts fail.

### BUG C: Whale detection rate-limited (429) — degraded ❌
**From error log** (20+ lines of):
```json
{"level":"warn","message":"Trades API returned non-OK","status":429}
```
Polymarket's `data-api.polymarket.com/trades` endpoint is returning 429 Too Many Requests. The system checks whale trades for every market with an odds change (~450/cycle × 10 markets each = thousands of requests). The rate limiting means whale detection is essentially non-functional for most markets.

**Impact**: `whales=18` in logs but most are likely false — previously cached whale data. Confidence scoring loses the `+10-20` whale bonus for most signals.

### BUG D: DB WAL file is 122MB — checkpoint needed ⚠️
`data/polysignal.db-wal` is 122MB. SQLite WAL mode accumulates writes until a checkpoint. With 10-min scan cycles inserting thousands of snapshots, the WAL grows unboundedly if auto-checkpoint isn't triggering. This will cause read performance degradation.

### BUG E: `alertsSent` metric is misleading ⚠️
In `scan-cycle.ts:94`:
```ts
alertsSent: signals.filter(signal => signal.confidence >= this.config.alertMinConfidence).length
```
This reports how many signals COULD be alerts, not how many HA pushes were actually sent. The log shows `alerts=39` but zero HA notifications may have been delivered. Makes debugging impossible.
